# 에니어그램 테스트 개선안

## 목적

이 문서는 현재 프로젝트의 문항 구성과 점수 계산 로직을 검토한 뒤, 결과 왜곡을 줄이고 제품 포지셔닝을 명확히 하기 위한 개선 방향을 정리한 것이다.

핵심 목표는 아래 3가지다.

- 구조적 편향을 제거한다.
- 결과를 "정확한 진단"이 아니라 "해석 가능한 성향 프로필"로 정직하게 제시한다.
- 저장된 기존 결과를 깨지 않으면서 `v2`로 안전하게 이행한다.

## 현재 상태 요약

현재 구현은 다음 파일을 중심으로 동작한다.

- `src/content/assessments/ko/v1.ts`
- `src/domain/assessment/scoring.ts`
- `src/domain/assessment/normalization.ts`
- `src/domain/assessment/mappings.ts`
- `src/app/results/[publicId]/result-snapshot-view.tsx`

현 구조는 모바일에서 빠르게 완료되는 짧은 테스트로서는 단순하고 구현 비용이 낮다. 반면, "에니어그램 검사"의 정확도를 기대하게 만드는 순간 문제가 생긴다.

## 확인된 문제

### 1. 구조적으로 1번 유형 편향이 발생한다

현재 문항은 유형별 2문항씩 총 18문항이고, 각 문항의 가중치는 질문 내용과 무관하게 동일한 규칙으로 자동 생성된다.

- 주유형: `[0, 1, 2, 4, 6]`
- 인접 유형: `[0, 1, 1, 2, 3]`
- 나머지 유형: `[0, 0, 1, 1, 1]`

이 구조에서는 아래와 같은 응답이 모두 전 유형 동점이 된다.

- 모든 문항 1점
- 모든 문항 2점
- 모든 문항 3점
- 모든 문항 4점
- 모든 문항 5점

그 뒤 `primaryType`은 동점일 때 `lowest-type-id` 규칙으로 정해지므로 항상 `1번`이 나온다. 이는 제품 신뢰도 관점에서 치명적이다.

### 2. 문항이 실제로는 "개별 문항"으로 작동하지 않는다

지금은 문항 하나하나가 별도 판별력을 가진다기보다, 각 문항에 미리 부여한 타입 라벨이 거의 모든 결과를 결정한다.

- 같은 유형에 속한 두 문항은 점수 구조가 완전히 같다.
- 질문 wording은 다르지만, scoring 상으로는 사실상 같은 카드 2장이다.
- 문항별 미세한 차이, 역문항, 혼동 유형 분리 기능이 없다.

즉, 현재는 "문항 기반 측정"이라기보다 "유형별 자기보고 체크리스트"에 가깝다.

### 3. wing 계산이 과도하게 단정적이다

현재 wing은 인접 두 유형 중 raw score가 높은 쪽으로 강제 결정된다. 동점이면 더 작은 번호를 선택한다.

- `1 -> 2`
- `3 -> 2`
- `5 -> 4`
- `9 -> 1`

처럼 구조적으로 한쪽 wing이 더 자주 선택될 수 있다.

실제 에니어그램에서는 wing이 뚜렷하지 않거나 양쪽 영향이 비슷할 수 있는데, 현재 구현은 이를 표현하지 못한다.

### 4. 성장/스트레스 방향은 "측정값"이 아니라 "이론 매핑"이다

`growthType`과 `stressType`은 실제 응답에서 계산되는 값이 아니라, 주유형이 정해진 뒤 고정 매핑으로 따라온다.

이 자체는 에니어그램 일반 설명과 맞는다. 다만 결과 화면에서 측정 결과처럼 보이면 오해의 소지가 있다.

### 5. 정규화 점수가 확률처럼 보인다

현재 정규화는 아래 식으로 계산된다.

`rawScore / totalRawScore`

이 방식은 9개 유형 점수의 합이 거의 100이 되므로 사용자가 이를 확률이나 비율처럼 오해하기 쉽다. 하지만 실제로는 모집단 기반 백분위도 아니고, 독립적인 척도도 아니다.

### 6. 테스트가 "현재 규칙 고정"에만 초점이 있다

현재 테스트는 구현이 의도대로 동작하는지는 잘 검증한다. 그러나 아래와 같은 도메인 품질은 검증하지 않는다.

- 균일 응답이 왜곡 없이 처리되는가
- 애매한 응답을 무리하게 한 유형으로 확정하지 않는가
- wing이 불명확할 때 이를 올바르게 표시하는가
- 결과 화면 문구가 측정 범위를 넘어서지 않는가

## 제품 관점의 판단

현재 범위와 모바일 UX 목표를 고려하면, `정밀 진단형 검사`로 가기보다 `짧지만 덜 왜곡된 해석형 검사`로 재설계하는 쪽이 현실적이다.

권장 포지셔닝은 아래와 같다.

- 유지할 것: 빠른 완료, 모바일 최적화, 공유 가능한 결과
- 낮출 것: "정확한 타입 판별"이라는 암묵적 기대
- 강화할 것: 해석 가능성, 근접 유형 비교, 결과 신뢰도 표시

참고로 Enneagram Institute의 RHETI는 144개의 강제선택 문항으로 구성된 별도 검사지이며, 9개 유형 전체 프로필을 제공한다. 현재 18문항 Likert 설문은 그 수준의 정밀도를 기대하기 어렵다.

## 권장 개선 방향

### 1. `v2`에서 문항 수를 늘리고 문항 설계를 바꾼다

최소 권장안은 총 36문항이다.

- 유형당 4문항
- 가능하면 유형당 6문항, 총 54문항까지 확장
- 한 유형 안에서도 서로 다른 측면을 다룬다

권장 문항 축은 아래 4가지다.

- 핵심 동기
- 주의 초점
- 방어 패턴
- 대인 반응

문항 작성 원칙은 아래와 같다.

- 사회적으로 바람직해 보이는 표현을 줄인다.
- "행동"만 묻지 말고 "왜 그렇게 하는지"를 드러낸다.
- 혼동이 잦은 유형쌍을 분리하는 문항을 의도적으로 넣는다.
- 일부 역문항을 포함해 무의식적 일괄 체크를 줄인다.

우선 분리 성능을 강화해야 할 대표 혼동쌍은 아래와 같다.

- 1 vs 6
- 2 vs 9
- 3 vs 8
- 4 vs 5
- 6 vs 9
- 7 vs 8

### 2. 자동 생성 가중치를 없애고 문항별 명시적 스코어링으로 바꾼다

`buildTypeWeights()` 방식은 제거하는 것이 좋다. 대신 각 문항이 어떤 유형을 얼마나 반영하는지 직접 정의해야 한다.

권장 방식은 `중심화 점수(centered scoring)`다.

- 1점: `-2`
- 2점: `-1`
- 3점: `0`
- 4점: `+1`
- 5점: `+2`

역문항은 부호를 반대로 적용한다.

예시 스키마:

```ts
type AssessmentQuestionV2 = {
  id: string;
  prompt: string;
  keyedType: EnneagramType;
  reverse: boolean;
  dimension: "motivation" | "attention" | "defense" | "interpersonal";
};
```

예시 계산:

```ts
const centered = question.reverse ? 3 - answer.value : answer.value - 3; // -2..2
rawScores[question.keyedType] += centered;
```

이 구조의 장점은 아래와 같다.

- 전 문항 `3점`은 전 유형 0점이 되어 자연스럽다.
- 균일 응답이 특정 유형으로 쏠리지 않는다.
- 역문항을 쉽게 추가할 수 있다.
- 유형별 문항 수가 달라도 보정이 가능하다.

### 3. 정규화는 "합 100" 방식 대신 독립 지표로 바꾼다

현재처럼 총합 대비 비율로 보여주면 확률처럼 오해된다. 각 유형 점수는 서로 독립적으로 해석되는 상대 강도 지표여야 한다.

권장 방식은 아래 둘 중 하나다.

- `-100 ~ 100` 범위의 centered index
- 이론적 최소/최대값 기준 `0 ~ 100`으로 독립 정규화

예시:

```ts
const minRaw = -2 * itemCountForType;
const maxRaw =  2 * itemCountForType;
const normalized = Math.round((((raw - minRaw) / (maxRaw - minRaw)) * 1000)) / 10;
```

이 경우 각 유형의 점수는 독립적으로 해석할 수 있고, 9개 합이 100이 되지 않으므로 확률처럼 보이지 않는다.

### 4. "낮은 신뢰도" 상태를 추가한다

가장 중요한 개선점 중 하나다. 모든 응답을 억지로 한 유형으로 확정하지 말아야 한다.

권장 상태값:

- `clear`
- `mixed`
- `insufficient_variance`

판정 기준 예시:

- 모든 응답이 동일하거나 분산이 매우 낮으면 `insufficient_variance`
- 1위와 2위 점수 차가 작으면 `mixed`
- 그 외에만 `clear`

결과 구조 예시:

```ts
type ResultStatus = "clear" | "mixed" | "insufficient_variance";
```

이 상태는 DB와 결과 UI에 모두 반영해야 한다.

### 5. wing은 "선택적"으로만 노출한다

wing은 무조건 표시하지 않는 쪽이 낫다. 아래 조건을 만족할 때만 노출하는 것을 권장한다.

- 주유형이 `clear`
- 인접 두 wing 후보 중 상위 후보가 일정 임계값 이상 우세
- 해당 wing 점수가 의미 있는 수준 이상

그 외 경우는 아래처럼 처리한다.

- `wingType: null`
- UI 문구: `날개는 뚜렷하지 않음`

즉, wing은 "후보"이지 "항상 계산되는 확정값"이 아니어야 한다.

### 6. 성장/스트레스는 참고 정보로 낮춰서 보여준다

성장/스트레스 방향은 계속 유지해도 된다. 다만 명확히 아래처럼 표현해야 한다.

- 현재 응답으로 측정된 별도 점수 아님
- 주유형 해석을 보조하는 이론적 연결선
- 결과 카드의 참고 섹션으로 이동

권장 문구:

`성장 방향과 스트레스 방향은 현재 응답에서 별도로 측정된 결과가 아니라, 주유형 해석을 돕기 위한 에니어그램의 일반적 연결선입니다.`

### 7. 결과 화면 카피를 더 정직하게 바꾼다

현재 카피는 대체로 절제되어 있지만, UI 구조상 `주 유형 / 날개 / 성장 방향 / 스트레스 방향`이 모두 같은 위상으로 보인다.

`v2`에서는 우선순위를 아래처럼 바꾸는 것이 좋다.

- 1순위: 주유형 후보
- 2순위: 근접 유형
- 3순위: 결과 선명도
- 4순위: 날개 후보
- 5순위: 성장/스트레스 참고 방향

권장 예시:

- `가장 가까운 유형 후보: 6번`
- `가까운 다른 후보: 1번, 9번`
- `이번 응답은 유형 간 차이가 크지 않아 혼합적으로 해석하는 편이 적절합니다.`

## 파일별 변경 권장안

### `src/content/assessments/ko/v1.ts`

- `buildTypeWeights()` 제거
- `questionFocusById` 제거
- 문항을 개별 정의 방식으로 변경
- `v2` 문항 세트 신규 추가

### `src/domain/assessment/types.ts`

- 문항 타입에 `keyedType`, `reverse`, `dimension` 추가
- 결과 타입에 `resultStatus`, `confidence`, `wingType | null` 추가

### `src/domain/assessment/scoring.ts`

- centered scoring 도입
- uniform/flat response 처리
- low-confidence 판정 추가
- wing optional 계산으로 변경

### `src/domain/assessment/normalization.ts`

- 총합 대비 백분율 방식 제거
- 유형별 독립 정규화로 교체

### `src/domain/assessment/mappings.ts`

- growth/stress 매핑은 유지
- 단, 결과 구조상 `measured`가 아니라 `theoretical` 성격으로 분리

### `src/app/results/[publicId]/result-snapshot-view.tsx`

- 결과 위계 재정렬
- wing이 없을 수 있는 UI 처리
- 성장/스트레스 문구를 참고 정보로 하향
- `정규화 점수 분포` 명칭을 `상대 강도 지표` 또는 `유형별 반응 지표`로 변경

### DB / 저장 구조

`assessment_results`에 아래 필드 추가를 권장한다.

- `result_status`
- `confidence_score`
- `interpretation_flags`

`wing_type`는 nullable로 바꾸는 것이 좋다.

기존 결과와의 호환을 위해 아래 원칙을 지킨다.

- `ko-enneagram-v1` 결과는 재채점하지 않는다.
- `ko-enneagram-v2`를 새 assessment version으로 추가한다.
- 공개 결과 페이지는 저장 시점의 `assessmentVersion`, `scoringVersion`, `copyVersion`을 그대로 따른다.

## 테스트 강화안

현재 테스트는 유지하되, 아래 케이스를 추가해야 한다.

### scoring 테스트

- 전 문항 동일 응답 시 `1번 고정`이 나오지 않는지
- 전 문항 `3점`일 때 `insufficient_variance`가 되는지
- top1/top2 격차가 작을 때 `mixed`가 되는지
- wing이 불명확하면 `null`이 되는지
- 역문항이 반대로 계산되는지
- 유형별 독립 정규화가 합 100을 강제하지 않는지

### definition 테스트

- 유형당 최소 문항 수 보장
- 역문항 비율 보장
- 혼동쌍 분리 문항 존재 여부 확인

### result UI 테스트

- 낮은 신뢰도 결과 문구 렌더링
- wing 없음 상태 렌더링
- 성장/스트레스가 참고 정보로만 표시되는지

## 단계별 실행안

### Phase 1. 왜곡 제거

목표는 "틀린 확정"을 멈추는 것이다.

- low-confidence 상태 도입
- wing optional 처리
- 결과 화면 카피 조정
- 성장/스트레스 표시 위상 조정

이 단계만으로도 가장 큰 신뢰도 문제는 줄일 수 있다.

### Phase 2. scoring `v2` 도입

- centered scoring 도입
- independent normalization 도입
- `assessmentVersion/scoringVersion/copyVersion` 갱신
- 신규 테스트 추가

이 단계부터 결과 구조가 실제로 개선된다.

### Phase 3. 문항 재작성

- 최소 36문항 구성
- 혼동쌍 분리 문항 작성
- 역문항 반영
- 내부 시뮬레이션으로 치우침 점검

### Phase 4. 해석 품질 검토

- 결과 카피 정비
- 한국어 표현 다듬기
- 지나친 단정 표현 제거
- 모바일 결과 화면 정보량 최적화

## 권장 우선순위

가장 현실적인 우선순위는 아래와 같다.

1. `1번 고정 편향` 제거
2. `low-confidence` 상태 추가
3. `wing 강제 확정` 제거
4. `정규화 방식` 교체
5. `문항 수 확대`와 `문항 재작성`

## 완료 기준

`v2` 개선이 충분히 됐다고 볼 수 있는 최소 기준은 아래와 같다.

- 균일 응답이 특정 유형으로 강제되지 않는다.
- 애매한 결과를 `mixed` 또는 `insufficient_variance`로 구분한다.
- wing은 불명확하면 비워 둘 수 있다.
- 성장/스트레스는 참고 정보로만 보인다.
- 정규화 점수는 합 100을 강제하지 않는다.
- 유형당 문항이 충분하고, 문항 간 판별 역할이 겹치지 않는다.
